## JSON command protocol (Java side)

You’re using **`Networking.java`** with console input. Here’s what it sends:

| Console Input | JSON Sent |
|---------------|-----------|
| `pan scan` | `{"type":"pan_scan"}` |
| `pan 45` | `{"type":"pan","angle":45}` |
| `forward 100` | `{"type":"forward","value":100}` |
| `stop` (single word) | `{"type":"stop"}` |

**ReconServer** (if used) is different: it turns the whole line into `type`:

- `forward 100` → `{"type":"forward 100"}` (one string, not `type` + `value`)

So first step: confirm which Java main you run — **Networking** or **ReconServer**.

---

## Common causes of "Unknown command" on the Pi

### 1. Command type string mismatch

Python is probably doing something like:

```python
if cmd["type"] == "move_forward":   # but Java sends "forward"
if cmd["type"] == "forward ":       # trailing space or typo
```

Java sends `forward`, `backward`, `pan`, `pan_scan`, `stop`, etc. Python must use exactly those strings (case-sensitive).

### 2. Type vs value vs angle

Java formats:

- `{"type":"forward","value":100}` → `value` is a number
- `{"type":"pan","angle":45}` → uses `angle`, not `value`
- `{"type":"stop"}` → only `type`, no extra fields

Python should:

- Check for `angle` only for `pan`
- Check for `value` only for movement commands that need a numeric argument
- Accept `stop` with no extra fields

### 3. Empty / malformed lines

`readline()` can return empty lines. Ignore them before parsing:

```python
line = sock.recv(...).decode().strip()
if not line:
    continue
```

### 4. Buffer / threading

`PrintWriter` with `autoFlush=true` should flush on `println`. If you ever use `print` or non-flushing writers on either side, some JSON lines might be delayed or stuck.

---

## 1. ACK messages → "Unknown command"

The server sends `{"type":"ack","timestamp":...}` after every Pi message (see `Networking.java` around line 171). The Pi receives these, but `execute_command()` has no case for `"ack"`, so they fall through to:

```python
else:

```

So every ACK is printed as "Unknown command". That’s likely what you’re seeing.

**Fix:** handle ACKs explicitly:

```python
elif ctype == "ack":
    pass  # Server acknowledgment, ignore
```

Place this before the `else:` in `execute_command`.

---

## 2. TCP message fragmentation

`sock.recv(1024)` can return partial data. A JSON line like `{"type":"forward","value":100}\n` might arrive as:

- First recv: `{"type":"forward","value`
- Second recv: `100}\n{"type":"ack"...`

Then `split("\n")` produces:

- `{"type":"forward","value` → invalid JSON → exception
- `100}` → invalid JSON → exception

Use a line buffer and only parse full lines:

```python
buffer = ""
while True:
    try:
        data = sock.recv(1024)
        if not data:
            continue
        buffer += data.decode("utf-8")
        while "\n" in buffer:
            line, buffer = buffer.split("\n", 1)
            line = line.strip()
            if not line:
                continue
            try:
                cmd_obj = json.loads(line)
                print(f"[AUTO] Received JSON: {cmd_obj}")
                execute_command(cmd_obj)
            except json.JSONDecodeError as e:
                print(f"[ERROR] Invalid JSON: {line!r} - {e}")
    except Exception as e:
        ...
```

---

## 3. JSON format match

Your Java and Python formats line up:

| Command   | Java sends                           | Python expects        |
|----------|---------------------------------------|------------------------|
| pan scan | `{"type":"pan_scan"}`                 | ✓ `pan_scan`          |
| pan 45   | `{"type":"pan","angle":45}`           | ✓ `pan` + `angle`     |
| forward 100 | `{"type":"forward","value":100}`   | ✓ `forward` + `value` |
| stop     | `{"type":"stop"}`                     | ✓ `stop`              |

---

- Add an `ack` handler in `execute_command` to stop ACKs being reported as "Unknown command".
- Add a line buffer in the receive loop so only complete lines are parsed and partial reads don’t cause parse errors.
